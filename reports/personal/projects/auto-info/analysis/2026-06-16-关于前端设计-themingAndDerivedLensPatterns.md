## 模式一：CSS 变量驱动的可切换主题系统

### 适用条件（判断是否该用此模式）
- 项目颜色/质感已（或可低成本收敛到）CSS 自定义属性。
- 需求是"换肤/多主题/明暗切换"，而非局部配色调整。
- 不希望引入 Tailwind 等框架做整页迁移（低风险、可叠加）。

### 机制（与具体项目无关）
1. **令牌真源单文件**：所有颜色/质感/字体令牌集中到一个 `_theme.css`，业务样式只引用 `var(--token)`，不出现字面色值。
2. **属性选择器切换**：`<html data-theme="x">` + `[data-theme="x"]{ --token: ... }`，JS 只改一个 attribute 即整站换肤。默认主题同时写进 `:root`，保证无 JS 时与原界面一致。
3. **令牌分层**：结构令牌（圆角/间距/字体族）放 `:root` 跨主题共享；颜色/质感令牌按主题覆盖。
4. **扩展令牌覆盖硬编码**：换肤失败的根因通常是散落的字面色（尤其深色主题下露白）。需补一组语义令牌：`surface` 多级（raised/sunken/pill）、`on-accent`、`input-border`、`warn` 三色、`shadow-color`、可选 `glow`/`backdrop` 装饰层。
5. **"提亮剂"陷阱**：`color-mix(..., #fff)` 这类用白色提亮的写法，深色主题会变白；统一改为跟随 `var(--surface)`，实现浅亮深暗自适应。
6. **shorthand 覆盖陷阱**：若令牌文件给 `body` 设了 `background-image`（网格/装饰层），后续 import 的样式表用 `background:`（shorthand）会清空它；改用 `background-color:` 单属性。
7. **持久化**：主题选择存 `localStorage`，启动时读取并 `setAttribute`，避免闪烁。

### 落地检查清单
- [ ] 全仓库搜 `#[0-9a-f]{3,6}` 与 `rgba(`，确认非令牌字面色已收敛（装饰性故意字面色除外，如主题预览 swatch）。
- [ ] 每套主题逐一切换，重点查深色：卡片底、表头、胶囊标签、输入框、阴影、SVG fill/stroke。
- [ ] 默认主题与重构前像素级一致。

## 模式二：派生透镜层（Derived Lens）

### 适用条件
- 已有数据展示完整，但用户认知负担高、抓不住重点。
- 想叠加一个"判断框架/聚合视角"，又不能破坏或替换既有展示。

### 机制
1. **纯函数派生层**：单独一个 `lib/<lens>.ts`，输入既有数据结构，输出聚合模型；无副作用、无请求、可单测。
2. **只派生不伪造**：所有字段从真实数据提取/归约；缺数据降级为空态由 UI 处理，绝不造假数据填充。
3. **分级与归约**：用既有信号（如严重度/时间/类别）做分桶、排序、计数、Top-N，把"全量列表"压缩为"可判断的少数维度"。
4. **叠加而非替换**：透镜组件置于原展示**之上/之前**，原组件原样保留，保证可回退、低风险。
5. **动作可执行**：聚合不止于展示，应派生指向具体操作的 action（跳转/打开/深读），把"看见"接到"行动"。

### 与判断框架结合
本项目用「发生了什么 / 影响是什么 / 能做什么」三问作为透镜骨架。该骨架可复用于任何"信息→决策"场景：第一栏=事实排序，第二栏=影响评估（分级+分布），第三栏=可执行动作。

## 模式三：前端信息密度处理层（去噪 / 折叠 / 评分）

### 适用条件
- 数据源含套话、模板句、近似重复、低信息量条目，拉高阅读成本。
- 后端清洗可能不彻底，或前后端分离、前端无法复用后端清洗逻辑（不同语言/不能 import）。
- 目标是"减负"而非改写：过滤、折叠、降权，不重写语义、不伪造内容。

### 机制（纯函数，独立可单测）
单独 `lib/<density>.ts`，三类能力可组合：
1. **去套话 + 去回声**：模板句正则集判废；摘要若是标题回声（归一化后被标题包含/前缀重合），取增量尾部或返回空。归一化 = 去标点/空白/大小写后比较。
2. **近似重复折叠**：归一化标题做相似度聚类（双字 gram 交集占比 + 子串覆盖），同组保留最高优先级者为代表，记录折叠数。**阈值取舍：判断类场景宁可漏合并也不误合并**——把不同事件揉成一条的危害 > 多看一条重复。本项目用 0.7，实测不同事件 sim≤0.18 零误伤。
3. **信息密度评分**：高密度锚点（数字/金额/百分比/机构/时间/因果动作词）加分，套话/过短减分，输出 0-100。供 UI 排序上浮或低分降权（弱化 opacity 而非隐藏，保留可读+可回溯）。

### 与派生层/UI 的接合
密度层接在模式二的派生层里：派生时 clean→dedupe→score，给条目挂 `density`/`dupCount`；排序兼顾业务优先级与密度；UI 显示"+N 同类"折叠标记、低密度弱化样式。三层分工清晰：`density`（无状态算法）→ `lens`（派生聚合）→ 组件（呈现）。

### 落地检查清单
- [ ] 相似度阈值用真实样例验证：不同事件不被误合并、近似事件正确合并。
- [ ] 低密度信号弱化而非删除，hover/展开可恢复，保证可追溯。
- [ ] 纯函数零副作用，可脱离 React 单测。

## 通用收尾经验（已验证）
- **构建环境平台陷阱**：跨平台沙箱（如 Linux）跑不了为另一平台（macOS）安装的原生二进制依赖（rollup/esbuild/swc）。验证时退到纯 JS 工具：TypeScript 编译器 `node node_modules/typescript/lib/tsc.js -p <tsconfig>` 可在任意平台做类型检查，无需原生二进制。
- **无 tsconfig 项目**：靠 esbuild 转译的项目常无 tsconfig 且从不类型检查，历史文件可能积累类型错误；新增改动做类型检查时，按"我的文件是否零错误"判断，而非全仓库零错误。
