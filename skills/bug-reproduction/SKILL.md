# Bug Reproduction

用这个技能处理 bug。目标是先复现或确认症状，再定位根因，最后做最小修复和回归验证。

## Trigger

当用户说以下任一类需求时使用：

- “报错了”“这个测试失败”“页面不对”“接口返回错”
- “帮我 debug”“定位原因”“修一下 bug”
- “为什么会这样”“复现一下”

## Workflow

1. **Capture Symptom**
   - 记录错误信息、触发步骤、输入、环境、版本、最近改动。
   - 如果用户只给截图或描述，先从可见线索提取可验证假设。

2. **Reproduce**
   - 优先运行最小命令、单测、页面路径或 API 请求。
   - 无法复现时说明缺少什么，并用日志/代码路径确认可能原因。

3. **Localize**
   - 从错误边界往内追：入口、参数、状态、依赖、核心逻辑、输出。
   - 区分根因、触发条件和表面症状。
   - 不急着大范围重构。

4. **Fix Minimally**
   - 修改最小必要代码。
   - 保留用户已有改动。
   - 若发现相邻问题，先标注，不顺手扩大范围。

5. **Add Regression Coverage**
   - 能加测试就加最窄回归测试。
   - 不能加自动测试时，写出手动复现和验证步骤。

6. **Verify**
   - 重新运行复现命令。
   - 再运行相关单测、类型检查、lint 或页面 smoke。
   - 确认错误不只是被隐藏，而是目标行为成功。

## Output

默认输出：

```markdown
## Root Cause

## Fix

## Verification

## Remaining Risk
```

## Quality Bar

- 不能只说“应该修好了”，必须说明验证信号。
- 如果只验证了错误态暴露，要明确说还没有端到端成功。
- 修复要尽量靠近根因，不用宽泛 fallback 掩盖问题。
