# Bug Reproduction

用这个技能处理 bug。目标是先复现或确认症状，再定位根因，最后做最小修复和回归验证。

## Trigger

当用户说以下任一类需求时使用：

- “报错了”“这个测试失败”“页面不对”“接口返回错”
- “flaky”“timeout”“hang 住了”“偶发失败”“CI 偶现失败”
- “帮我 debug”“定位原因”“修一下 bug”
- “为什么会这样”“复现一下”

## Workflow

1. **Capture Symptom**
   - 记录错误信息、触发步骤、输入、环境、版本、最近改动。
   - 如果用户只给截图或描述，先从可见线索提取可验证假设。

2. **Reproduce**
   - 优先运行最小命令、单测、页面路径或 API 请求。
   - 无法复现时说明缺少什么，并用日志/代码路径确认可能原因。
   - 对 flaky、timeout、hang 或并发问题，记录失败频率、并行度、环境差异和等待条件；不要先加 sleep 或扩大 timeout。

3. **Root Cause Investigation**
   - 没有根因证据前不提修复方案；先读完整错误、栈、日志和最近 diff。
   - 从表面症状沿调用链向上追：谁传入了坏值、状态在哪里改变、哪个边界丢了语义。
   - 多组件链路要在边界处补临时诊断：输入、输出、配置、状态和下游响应，先证明断点在哪一层。

4. **Pattern Analysis**
   - 找同仓库可工作的相似实现或历史测试，对比差异。
   - 有参考实现、官方示例或历史成功路径时，先完整阅读相关片段，再列 working/broken 差异；不能只凭印象复制一段近似代码。
   - 区分根因、触发条件、防御缺口和表面症状；不要把 fallback 当根因修复。
   - 异步或并发问题优先等待真实条件，例如事件、状态、计数、文件存在；除非目标就是 timing 行为，不用任意 sleep 掩盖 race。
   - 任意 sleep 只有在 debounce、throttle、固定 tick、重试间隔等真实 timing 行为是目标时才允许；必须先等触发条件，再写明等待时长来自哪个机制。
   - 如果系统性排查后仍只能指向外部、环境或时序因素，先列出已排查边界、证据和仍未知项；不能为了收口编造 root cause。

5. **Hypothesis Test**
   - 每次只提出一个可验证假设：`我认为 X 是根因，因为 Y`。
   - 用最小改动、临时日志或窄测试验证假设。
   - 记录假设、验证动作、输出和结论；失败的假设要明确标为证伪，不能继续在其上叠补丁。
   - 如果连续 3 次修复尝试失败，停止叠补丁，重新审查架构、边界假设、测试前提或需求理解；不要尝试第 4 个局部补丁。

6. **Fix Minimally**
   - 修改最小必要代码。
   - 保留用户已有改动。
   - 若发现相邻问题，先标注，不顺手扩大范围。
   - 线上应急只能先做可逆 mitigation；必须明确标注“止血/降级/隔离”，不能把未定位根因的 mitigation 称为修复完成。

7. **Add Regression Coverage**
   - 能加测试就加最窄回归测试。
   - 行为 bug 默认先证明测试在修复前会失败，再证明修复后通过；若无法自动化，写清手动复现与失败/成功信号。
   - 不能加自动测试时，写出手动复现和验证步骤。

8. **Verify**
   - 重新运行复现命令。
   - 再运行相关单测、类型检查、lint 或页面 smoke。
   - 确认错误不只是被隐藏，而是目标行为成功。

## Output

默认输出：

```markdown

## Verification

```

## Quality Bar

- 不能只说“应该修好了”，必须说明验证信号。
- 如果只验证了错误态暴露，要明确说还没有端到端成功。
- 修复要尽量靠近根因，不用宽泛 fallback 掩盖问题。
- 不把“看起来很可能”当根因；根因必须有复现、追踪、对比或最小假设验证支撑。
- 没找到根因时要明确说“根因未闭环”，并给出已排查证据、临时处理、观测条件和下一步，而不是把环境/外部/时序猜测包装成 root cause。
