# Source-Grounded Research

用这个技能做有来源支撑的调研，不把模型记忆当作最新事实。目标是给用户一份可复用、可追溯、能区分事实与判断的整理。

## Trigger

当用户说以下任一类需求时使用：

- “结合业界资料”“联网查一下”“最新资料”“调研”“竞品分析”“论文/标准/官方文档”
- 需要直接引用来源链接、发布日期、版本、规格、价格、政策或公司产品信息
- 需要判断某个技术方向、行业做法或方案取舍是否仍然成立

## Workflow

1. **Clarify Scope Only If Needed**
   - 如果用户已经给出主题和交付物，直接开始。
   - 只有在主题过宽、来源边界不清或结论会影响高风险决策时，才问一个关键问题。

2. **Build Search Plan**
   - 先列出 3 到 6 个核心问题。
   - 优先查官方文档、标准/spec、release notes、论文、可信工程博客和一手案例。
   - 社区讨论、媒体报道和二手总结只能作为信号，不能单独作为事实依据。

3. **Collect Sources**
   - 对快速变化领域必须联网核对。
   - 记录每个来源的标题、URL、发布日期或访问日期、可信度类别。
   - 对同一事实尽量找两个独立来源，冲突时以官方/原始来源优先。

4. **Extract Claims**
   - 每条重要结论拆成 `事实`、`推断`、`建议` 三类。
   - 事实必须能回到来源；推断要说明依据；建议要说明取舍。
   - 不确定处明确写“未确认”“资料不足”或“截至当前来源未看到”。

5. **Synthesize**
   - 先给结论，再给判断依据。
   - 聚合同类信息，不按来源逐篇堆摘要。
   - 如果用户要方案，输出可执行建议；如果用户要资料，输出结构化材料清单。

6. **Validate**
   - 检查是否引用了过时资料。
   - 检查是否把厂商营销口径当成独立事实。
   - 检查是否遗漏了反例、限制条件、定价/版本/地区等关键边界。

## Output

默认使用以下结构：

```markdown
## 结论

## 关键事实

## 行业做法 / 资料整理

## 判断与取舍

## 风险与未确认点

## 来源
```

## Quality Bar

- 每个关键判断至少能指向一个来源。
- 不用“业界普遍认为”这类空泛说法，除非给出足够证据。
- 对最新事实写明核对日期。
- 保持短结论和可追溯证据并存。
