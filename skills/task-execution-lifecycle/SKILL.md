# Task Execution Lifecycle

本技能定义用户发布任务后的执行闭环。目标是让主 Agent 主动推进和验证，同时按风险控制读取、review、同步和沉淀成本，不让完整流程吞掉小任务。

## Trigger

出现以下情况时使用：

- 用户发布非机械的实现、重构、UI、架构、协议、数据、工具链或跨模块任务。
- 用户要求“执行修改”“do it”“开始做”“继续”。
- 用户在任务开始时引用 agent-hub README；引用一次即可，后续必要时自动使用相关规则和 Skill。
- 任务进入收尾、准备提交、准备交用户审查阶段。

如果用户明确说“只分析/不要改代码/先别实现”，只执行分析和计划部分。typo、单点文档、小范围机械改动可走轻量路径，不加载本技能的长 reference。

## Execution Budget

- **轻量路径**：小修、小文档、单文件或机械调整。只读直接相关文件，确认显而易见的不变量，完成最小修改，跑 `git diff --check`、引用搜索或最窄命令；不自动创建计划、subagent、main sync、报告或完整 review。
- **标准路径**：普通实现、bugfix、UI 调整和局部重构。冻结 3-7 条关键不变量，按最小可验证闭环实现，做定向测试、UI/API smoke 或 diff 边界审查。
- **高风险路径**：接口/权限/安全/DB/生成物/runtime/tool/gateway/跨仓库/跨 agent、反复失败、准备 commit/PR/merge、用户要求研究或质量审查。按需读取 references，加入测试 Agent / review 视角，扩大验证和边界检查。
- 每次升级执行面时先问：这个动作能防止哪类返工或风险？答不上来就保持当前路径。

## Operating Rules

- 用户引用 hub 一次后，本轮任务后续默认持续适用 hub 规则；必要时可主动读取、引用 hub。更新或重构 hub 只在明确 `receive`、常驻任务的后续触发、定时/自动化触发、用户要求复盘，或出现反复且可复用的高风险流程缺口时执行。
- 规则更新、自我提升、报告沉淀是按触发执行的质量动作；除非用户当前目标就是整理 hub，否则不要让规则维护抢占实现闭环。
- 每次任务都先找准目标定位：这是管理态、创建态、运行态、调试态、协议态、数据态还是发布态；定位不清时先冻结 3-7 条不变量。
- coding 设计默认符合用户偏好：架构清晰、分层明确、交互合理、优雅、美观、影响可控，不碰预期之外内容。
- 多轮验收要收敛历史已知问题：过往对话中用户指出过、hub 已沉淀、历史验收失败或既有规则标记过的类似问题不得反复出现；若仍有疏漏，应尽量是新问题而不是同一类失败。
- 借鉴 Superpowers 的“证据优先”原则：任何完成、修好、通过、可用、已打通的结论，必须有本轮新鲜验证证据支撑；没有跑验证就只能报告当前推断或缺口。
- 对高风险、多文件或多 agent 任务，才使用“规格符合性 -> 代码质量/体验质量”的两段 review；轻量任务只做 readback 或 diff 自检。
- 派发 agent、工具任务或 review 视角时，按任务复杂度选择足够但不过度的能力；如果阻塞源于推理或上下文不足，升级能力或缩小任务，不原样重试。
- 当任务卡住、失败信号没有变窄、复杂 bug 修完后仍缺把握，或重构前缺 baseline 证据时，先按 `code-review` 的 Fresh Perspective Review Gate 组织目标、diff、证据缺口和已证伪假设；不要继续叠补丁。
- 子 agent 或工具执行者提出问题时，先判断是缺上下文、scope 冲突、计划缺口还是普通实现选择；补 brief、拆任务或问最小阻塞问题后再继续，不让执行者猜。
- 对长任务、多文件功能或可能与用户本地改动交错的任务，先识别当前 branch、worktree/宿主隔离状态和 dirty worktree；优先使用宿主已提供的隔离环境。只有用户同意且当前工具没有原生隔离能力时，才考虑手动 git worktree；不要为小修或文档整理制造额外 worktree。
- 在 `main`、`master`、默认分支或受保护分支上开始非平凡实现前，先确认用户是否明确要求在当前 checkout 维护；否则优先使用宿主隔离、明确 feature branch 或请求最小确认。用户明确指定当前仓库做规则/文档维护时，可以原地编辑，但收尾必须说明 branch、diff 边界和未提交状态。
- 当用户提供已有计划、spec、checklist 或要求“按计划执行”时，先批判性读取计划：确认目标、文件、步骤、验证和风险是否足够执行；若计划缺口会导致误改，先补计划或提出最小阻塞问题。计划可执行后连续推进，不在每个小步骤后反复问“是否继续”。

## Workflow

1. **Task Positioning**
   - 读取 hub 和项目历史文档中与任务直接相关的规则、报告、协议；没有直接风险信号时不扫全量历史。
   - 明确目标、非目标、owner、数据源、状态语义、影响面和验收信号。
   - 对需要隔离的任务，记录当前分支、是否已在 worktree/托管 workspace、未提交改动是否与任务重叠；发现重叠或不清楚时先收敛边界，不用 stash/reset 绕过。
   - 如果任务来自已有计划，先审查计划是否有占位、断链、过大步骤、验证缺口或与当前代码不符；计划不可靠时先修正计划再执行。
   - 对 UI/交互/架构类任务，先判断场景性质，再选择布局或分层，不先动手拼补丁。

2. **Design And Implementation**
   - 沿用现有架构和组件；新增抽象必须降低复杂度或匹配已有模式。
   - UI 先稳定信息结构、动作归属、状态反馈和视觉重量，再做细节样式。
   - 后端/API/tool/runtime 先保证单一真实链路、标准回执、错误态和验证信号。
   - 控制影响面：不顺手改无关模块、不清理用户未授权内容、不把调试分支长期化。
   - 行为变化、bugfix 和核心逻辑默认配套测试或等价验证；能测试先行时先写失败测试，再最小实现。

3. **Iterative Acceptance**
   - 根据用户反馈迭代，先修阻塞体验、行为 bug 和历史已知旧类问题。
   - 每轮迭代后检查：是否解决用户刚指出的问题，是否再次引入既往被指出过的问题、hub 已沉淀的反模式或历史验收失败类型，例如布局跳动、按钮漂移、数据顺序变化、mock/fallback 掩盖真实链路等。
   - 对 UI 任务，必要时用浏览器或截图验证真实页面，而不是只靠类型检查。

4. **Risk-Based Review And Refactor**
   - 功能完成后按风险 review 本轮改动：轻量任务做 diff/readback，自查范围和验证；标准任务检查架构、状态/错误、UI 信息层级、测试缺口和影响面；高风险任务再做两段 review。
   - 先检查是否满足用户目标和计划/规格，再检查代码质量、边界、可维护性和体验问题；不要用代码质量 review 代替规格符合性 review。
   - 多任务、并行/subagent 或分阶段计划全部完成后，做一次最终整合 review：跨任务接口一致性、重复/冲突修复、端到端验收和 diff 边界都要重新看；不能只把每个子任务的局部通过相加成整体完成。
   - 如果 review 发现不合理设计，应做最小必要重构；避免过度重构和机会主义清理。
   - 对可复用经验只做轻量候选判断；满足 `receive`、用户复盘、定时触发、反复缺口或高风险稳定原则时，才执行 `design-principle-library` 或相关 Skill 的更新，并简短告知但不中断主任务。

5. **Branch / Main Sync Gate**
   - 只有用户要求 `git update`、准备 commit/PR/merge、长期分支需要集成、或目标基线变化会影响本轮结论时，才按项目 git 规则更新 main/目标基线。
   - 合并后重新跑最窄有效验证，确认本轮改动仍成立。
   - 不使用破坏性 reset/checkout 覆盖用户改动。

6. **Prepare Commit Gate**
   - 只有用户要求提交、暂存、PR、合并，或任务明确进入分支收尾时，才准备 commit message 和完整 staged/unstaged/base-head diff 审查。
   - 普通交付先做与风险匹配的 diff 边界检查，确认没有明显越界；高风险或边界不清时再展开完整审查。
   - 在声明完成前重新运行最窄有效验证并读取输出；确认命令覆盖目标、不是 0 tests/skipped/no-op/cached/旧服务结果。旧输出、局部迹象、agent 口头成功或“应该没问题”都不算完成证据。
   - 对有 spec、plan、issue、验收标准或多轮用户纠偏的任务，先把要求和非目标列成 checklist，逐条绑定证据；不能用“测试通过”替代 requirements trace。
   - 若用户要求可直接 commit；否则仅在有价值时报告建议 commit message 和文件范围。

7. **Hand Off**
   - 最终交付说明按实际执行面裁剪：完成内容、验证结果、diff/范围状态和残余风险；只有做过 review、main sync 或 commit 准备时才汇报这些栏目。
   - 默认保留当前分支/工作区给用户审查；只有用户明确要求时才执行 commit、push、PR、merge 或 discard。
   - 如果用户要求结束分支工作，先确认验证通过、diff 边界和当前环境，再给出有限且按环境裁剪的结构化选项：保留审查、提交、创建 PR、合并、丢弃。不要用开放式“下一步怎么办”替代可判断选项；丢弃工作必须二次确认，不自动删除分支、worktree 或未提交内容。
   - 如果必要验证失败、未跑或证据不足，先停在修复、继续验证或保留审查；不要把 commit、PR、merge 或 discard 包装成可选收尾。
   - 明确哪些 hub 更新或规则沉淀已完成，哪些没有沉淀及原因。

## Output Shape

实现型任务收尾默认按实际执行面裁剪，可包含：

- `完成内容`
- `验证`
- `Review / 范围自检`
- `Main / 合并状态`（仅实际执行时）
- `建议 Commit`
- `待你审查`

中途状态更新保持短：当前在做什么、是否越界、下一步是什么。

## Quality Bar

- 不把“能跑”当“做好”；必须同时检查功能、代码设计、交互体验、影响面和验收标准。
- 不让规则维护、文档整理或外部研究拖住主任务；它们服务主线，不替代主线。
- 不把历史上用户纠偏过、hub 已沉淀过或验收失败过的同类问题再次作为已知缺陷交付。
- 不提交或建议提交前，必须完成 diff 边界审查和最窄有效验证。
- 必要验证失败或未跑时，不进入分支集成选项；只能报告失败、继续验证、修复或保留给用户审查。
- 不用“Done/修好了/通过了”一类结论替代证据；最终状态必须区分已验证、部分验证、未验证和阻塞层。
- 不用“Great/Perfect/搞定/没问题了”这类积极语气暗示未验证成功；中途更新也只陈述事实状态、验证缺口和下一步。
